Supabase 文件上传测试

问题诊断

错误信息: "new row violates row-level security policy"
可能的原因:
  • 存储桶的 RLS（行级安全）策略配置不正确
  • 用户认证状态问题
  • 存储桶权限设置问题
  • 策略条件过于严格

解决方案

步骤 1: 运行 RLS 修复脚本

在 Supabase SQL 编辑器中运行以下脚本：

-- 1. 删除现有策略 DROP POLICY IF EXISTS "允许所有人读取 temp_1 文件" ON storage.objects; DROP POLICY IF EXISTS "允许认证用户上传到 temp_1" ON storage.objects; DROP POLICY IF EXISTS "允许认证用户更新 temp_1 文件" ON storage.objects; DROP POLICY IF EXISTS "允许认证用户删除 temp_1 文件" ON storage.objects; -- 2. 创建简化策略 CREATE POLICY "temp_1_select_policy" ON storage.objects FOR SELECT USING (bucket_id = 'temp_1'); CREATE POLICY "temp_1_insert_policy" ON storage.objects FOR INSERT WITH CHECK (bucket_id = 'temp_1'); CREATE POLICY "temp_1_update_policy" ON storage.objects FOR UPDATE USING (bucket_id = 'temp_1') WITH CHECK (bucket_id = 'temp_1'); CREATE POLICY "temp_1_delete_policy" ON storage.objects FOR DELETE USING (bucket_id = 'temp_1');

步骤 2: 验证存储桶配置

在 Supabase Dashboard 中检查：

  • Storage → temp_1 是否存在
  • 存储桶是否为 "Public"（公开）
  • 文件大小限制是否合适
  • Policies 页面中策略是否正确应用

步骤 3: 测试用户认证

确保用户已正确登录：

// 检查用户认证状态 const { data: { user } } = await supabase.auth.getUser(); console.log('当前用户:', user);

备用方案

方案 A: 完全禁用 RLS（仅测试）

-- 临时禁用 RLS（测试后记得重新启用） ALTER TABLE storage.objects DISABLE ROW LEVEL SECURITY; -- 测试完成后重新启用 ALTER TABLE storage.objects ENABLE ROW LEVEL SECURITY;

方案 B: 使用管理员 API 密钥

在环境变量中使用服务角色密钥：

NEXT_PUBLIC_SUPABASE_URL=https://ptdutdjpifyjypmqnedc.supabase.co SUPABASE_SERVICE_ROLE_KEY=your_service_role_key_here

注意：服务角色密钥有完全访问权限，仅用于测试

方案 C: 检查存储桶所有者字段

Supabase Storage 可能需要设置 owner 字段：

// 在上传时设置 owner const { data, error } = await supabase.storage .from('temp_1') .upload(filePath, file, { cacheControl: '3600', upsert: true, // 可能需要手动设置元数据 metadata: { owner: user.id } });
